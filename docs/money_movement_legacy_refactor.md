# Money-Movement Legacy Refactor

## Overview

This document outlines the comprehensive refactor of the money-movement legacy system using the enhanced LLM prompts for data science and engineering workflows. The refactor focuses on modernizing legacy code while maintaining data integrity and implementing automated quality controls.

## Migration Path

### Phase 1: Code Output Optimization ✅
**Objective**: Prevent large code blocks from cluttering chat interactions
- Modified `generate_code.md` to use `cursor.write_file()` for code blocks > 40 lines
- Enhanced `validate_risk.md` to use concise 5-row Markdown tables instead of verbose JSON
- Created validation script `scripts/simulate_prompt.py` to ensure all prompts respect 40-line limit
- **Result**: All 13 prompts now produce concise, actionable output

### Phase 2: User Approval Checkpoints ✅
**Objective**: Ensure user consent before executing code generation
- Added approval checkpoints to `context_gap.md` and `generate_code.md`
- Standardized checkpoint message: *"Here's the draft plan. Respond **Proceed** to continue or **Revise** to change course."*
- **Result**: No code generation proceeds without explicit user approval

### Phase 3: Build/Test/Reconciliation Chain ✅
**Objective**: Create seamless automated testing and validation workflow
- **NEW**: `run_tests.md` prompt - Executes `dbt build --select {{ models }}` then `pytest tests/prompt_formatting`
- **Enhanced**: `reconcile.md` prompt - Calls Preset MCP: `preset.reconcile('{{ models }}')` with Markdown diff tables
- **Chain flow**: `getRunDbtPrompt` → `getRunTestsPrompt` → `getReconcilePrompt`
- **Result**: Comprehensive validation before deployment

### Phase 4: Auto-Publish Path ✅
**Objective**: Streamline PR creation and deployment process
- After successful reconciliation, automatically chains to `getCreatePRPrompt` → `getMergeGuardPrompt`
- **Default PR Title**: "♻️ Money-Movement Legacy Refactor – generated by MCP"
- **Auto-chain conditions**:
  - 🟢 ALL GREEN: Auto-chain immediately
  - 🟡 WARNINGS: Prompt user review
  - 🔴 ISSUES: Block auto-chain, require manual fix

### Phase 5: Macro & Style Guardrails ✅
**Objective**: Enforce code quality and consistency standards
- **Max 300 LOC per macro** - Break large macros into focused components
- **2-space indentation** - Consistent with project standards
- **Required doc-blocks**: `/** Purpose, Args, Returns, Example */` at top of every macro
- **Lint validation**: All code must pass `dbt deps && dbt compile` before commit
- **Jinja compliance**: All Jinja code passes linting checks

## Code Quality Standards

### Macro Documentation Template
```jinja2
/**
 * Purpose: Brief description of what the macro does
 * Args: 
 *   - arg1: Description of first argument
 *   - arg2: Description of second argument  
 * Returns: Description of what the macro returns
 * Example: {{ macro_name('param1', 'param2') }}
 */
{% macro macro_name(arg1, arg2) %}
  -- Implementation here (max 300 LOC)
{% endmacro %}
```

### SQL Formatting Rules
- Leading commas on column lists
- 2-space indentation (not 4-space)
- Meaningful table aliases (not single letters)
- Capitalized SQL keywords
- CTE documentation for complex logic

### Validation Pipeline
1. **Code Generation**: Enforced guardrails in `generate_code.md`
2. **Compilation Check**: `dbt deps && dbt compile` validation
3. **Test Execution**: Combined dbt build + pytest formatting tests
4. **Schema Reconciliation**: Automated diff analysis with Preset
5. **PR Creation**: Auto-generated with comprehensive documentation

## Troubleshooting

### Common Issues
1. **Approval checkpoint not triggered**: Check prompt file has USER APPROVAL CHECKPOINT section
2. **Code exceeds 40 lines in chat**: Verify generate_code.md uses cursor.write_file() logic
3. **Auto-chain blocked**: Review reconciliation results for 🔴 issues needing resolution
4. **Compilation failures**: Ensure all macros have proper Jinja syntax and doc-blocks
