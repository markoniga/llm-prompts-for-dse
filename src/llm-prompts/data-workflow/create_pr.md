# Create PR Prompt

## Purpose
This prompt helps generate comprehensive GitHub pull requests for dbt model changes. It creates well-structured PR titles, descriptions, and metadata that follow project conventions and provide clear context for reviewers.

## Usage
Use this prompt after code changes have been tested and are ready for review, to create a pull request that effectively communicates the changes and their impact.

## Input Context
- The code changes made
- Test results
- Project conventions for PRs
- Business context for the changes
- Affected models and dependencies
- Performance implications
- Migration considerations

## Output Format
```json
{
  "pr_metadata": {
    "title": "PR title following project conventions",
    "branch_name": "feature/branch-name",
    "base_branch": "main|development|other",
    "labels": ["label1", "label2"],
    "reviewers": ["@username1", "@username2"],
    "priority": "low|medium|high|urgent",
    "estimated_review_time": "10 minutes|30 minutes|1 hour|etc."
  },
  "pr_description": {
    "summary": "Brief summary of the changes",
```

## Instructions

1. **Analyze the changes**
   - Understand the nature and scope of the changes
   - Identify affected models and dependencies
   - Determine the business impact
   - Consider performance implications
   - Identify any migration considerations

2. **Create a descriptive title**
   - **Default for auto-generated PRs**: "🔄 Refactor / Cleanup – auto-generated by MCP"
   - Follow project conventions for PR titles
   - Include the type of change (e.g., feat, fix, perf)
   - Keep it concise but informative
   - Include ticket/issue reference if applicable

3. **Write a comprehensive description**
   - Summarize the changes clearly
   - Explain the motivation behind the changes
   - Provide technical details about the implementation
   - Describe how the changes were tested
   - Include performance impact information
   - Note any dependencies or migration considerations

4. **Create a checklist**
   - Include standard project checklist items
   - Add specific items relevant to the changes
   - Mark items as completed or pending
   - Add notes for items that need explanation

5. **Prepare release notes**
   - Describe user-facing changes
   - Document internal changes
   - Highlight any breaking changes
   - Specify required actions

6. **Develop a deployment plan**
   - Outline deployment steps
   - Create a rollback plan
   - Specify verification steps
   - Consider timing and dependencies

7. **Format the PR content**
   - Use Markdown formatting for readability
   - Follow project conventions for PR structure
   - Include appropriate headings and sections
   - Add links to relevant resources

## Safety Guardrails
- Always prioritize clarity and completeness in PR descriptions
- Include all relevant information for reviewers to understand the changes
- Consider the impact on downstream models and processes
- Provide clear deployment and rollback instructions
- Include appropriate testing information and results
- Document any special values or patterns introduced
- Consider performance implications of changes
- Highlight any breaking changes or required actions
- Follow project conventions for PR structure and formatting
- Include appropriate reviewers based on the changes made
